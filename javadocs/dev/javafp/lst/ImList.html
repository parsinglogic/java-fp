Package dev.javafp.lst

Interface ImList<A>

All Superinterfaces:
HasTextBox, Iterable<A>, Serializable
All Known Implementing Classes:
ImLazyList
public interface ImList<A> extends Iterable<A>, Serializable, HasTextBox

A mainly lazy functional list

Lazy

When you invoke map fn on a ImList , it will not do the eager thing of iterating over the items, applying fn and returning a new ImList .

Instead, it creates a ImMappedList that is a wrapper around the original list. When you invoke head() on this new ImList it will apply the function fn to the head of the wrapped ImList and return it. If you invoke tail() then it creates a new ImMappedList on the tail of the wrapped list.

Most of the standard functions on ImList s are implemented in this way.

To be fair, not all ImList s are completely lazy. Sometimes, a function has to do some work to determine if the ImList it returns is empty (and so it must return the ImEmptyList)

This means that we can handle infinite ImList s quite easily in the traditional lazy functional style.

However, this approach does pose some problems

  • equalsList(dev.javafp.lst.ImList<? extends A>) and size() on an infinite ImList will return ‚ä• (ie they will throw a stack overflow)

  • equalsList(dev.javafp.lst.ImList<? extends A>) and size() on a long ImList will be expensive.

  • a ImList doesn't know if it is infinite or not. Some lists will know this but others will be uncertain.

For most lists that get created in real programs, equalsList(dev.javafp.lst.ImList<? extends A>) and size() will be fine.

In fact hashCode(int) doesn't need to go right to the end of the ImList . We could stop after 10, say. Let's do that.

Most infinite lists will exist only briefly before having take(int) or drop(int) applied to them.

I have decided not to worry about equalsList(dev.javafp.lst.ImList<? extends A>) on infinite lists. I did dally with preventing equalsList(dev.javafp.lst.ImList<? extends A>) and size() on infinite lists - but I came to the conclusion that the extra faff caused by doing this was not worth it.

flush() will convert a ImList to a ImListOnArray (not ImListOnPrimitiveArray)

ImList the interface is the important type - not the implementations

The code that uses ImList will always use the ImList type - under the covers, there are many different implementations of ImList - but these are mainly invisible to the client code.

For example equalsList(dev.javafp.lst.ImList<? extends A>) only checks if the contents of the lists are the same - it does not matter what the underlying implementing class happens to be. This is similar to what Java does in List.equals(java.lang.Object)

  1. take - creates a ImConsList

Singleton ImEmptyList

I have decided to insist on having a singleton empty ImList . This means that some ImList functions have to do some extra work to determine which type of ImList to return - for example filter(dev.javafp.func.Fn<A, java.lang.Boolean>) might have to traverse the whole ImList before it works out that it should return the ImEmptyList.

The code is much simpler with singleton empty lists though.

To recurse or not?

For a number of these functions, we could implement by

  • recursive algorithms or

  • we could get an iterator on the underlying ImList and do the old-fashioned iteration thing.

Not sure what the best policy is.

ImLists can contain nulls

      default boolean equalsList(ImList<? extends A> other)

      Compares other with this for equality. Returns true if and only if the specified object is also a ImList , both lists have the same size, and all corresponding pairs of elements in the two lists are equal .

      e1 and e2 being equal means 

      
 e1.equals(e2) == true

      In other words, two ImLists are defined to be equal if they contain equal elements in the same order.

      In other other words - this is very similar to the standard Java List equals . I can't have a "normal" equals method that overrides equals(Object a) because that is not allowed in Java.

      I have put an equals method on the two subclasses of this class and they delegate here

    • upCast

      default <U> ImList<U> upCast()

      Allows ImList<U> to be cast to ImList<T> (where U is a subtype of T) without generating a compiler error - or even a warning.

      In general, Java does not regard this cast as type safe because of the "cuckoo in the nest" problem:

      The cuckoo in the nest problem

      Let's say we have a 'List' of 'Strings' - an ordinary mutable Java List 

      
 List<String> strings = new ArrayList<String>(...);

      Should you be able to make this assignment (an implicit cast)? 

      
 List<Object> nest = strings;   // List<String> is being implicitly cast to List<Object>

      At first glance this looks reasonable. Since every object in nest is a String it is also an Object - so we can get an element from nest and use a method that is defined in Object and there shouldn't be a problem.

      Indeed this would be ok.

      The problem is that Java Lists are mutable .

      So we could now do this: 

      
 Object cuckoo = new Object();
 nest.add(cuckoo);

      Again, at first glance, this seems ok too. nest is declared as a list of Object s, that cuckoo you added was an Object , so what's the problem?

      Well, the problem is with the List we started with - strings .

      strings and nest point to the same List . The List that we just added cuckoo to.

      So, from the point of view of strings , we now have a List of String s that contains one element that actually isn't a String . 

      
 strings.get(0).length();       // java.lang.ClassCastException: java.lang.Object cannot be cast to java.lang.String

      So Java does not allow you to cast a List of Strings to a List of Object s.

      In fact, Java goes even further. If you declare any class as having a type variable A (say), then it assumes that you are going to be storing an object of type A in instances of that class so if you had: 

      
 Wibble<String>  stringWibble = new Wibble(...);

      then Java will try to prevent you from doing this: 

      
 Wibble<Object> objectWibble = stringWibble;

      on the basis that a similar problem to that of List s will occur.

      java_fp has ImList which Java treats just like a Wibble - or any other class with a type parameter. It will prevent you saying this: 

      
 ImList<String> strings = ImList.on(...);
 ImList<Object> nest = strings;

      But, in java_fp , since ImList s are immutable , it is always safe to (implicitly) cast them in this way. You can't add a Cuckoo to nest because - well - you can't add anything to a ImList - in the sense of mutating it.

      Having established that it is safe to cast in this way, you will find that it is surprisingly difficult to actually do it without the Java compiler getting very cross.

      Let's look at an example.

      Assuming ints and numbers declared like this: 

      
 ImList<Integer> ints;
 ImList<Number>  ns;

      Let's see what the Java compiler has to say when we naively try various approaches (These error/warnings are from the Eclipse Java compiler):

      This obviously produces an error: 

      
 ns = ints;                                      => Type mismatch: cannot convert from ImList<Integer> to ImList<Number>

      This also produces an error: 

      
 ns = (ImList<Number>) ints;                        => Cannot cast from ImList<Integer> to ImList<Number>

      In desperation (how many times have we all been here?) we try this: 

      
 ns = (ImList<? extends Number>) ints;               => Type mismatch: cannot convert from ImList<capture#1-of ? extends Number> to ImList<Number>

      This (somewhat recondite) cast is allowed, merely generating a warning: 

      
 ns = (ImList<Number>) (ImList<? extends Number>) ints; => Type safety: Unchecked cast from ImList<capture#1-of ? extends Number> to ImList<Number>

      For your comfort and safety, we have provided a function, upCast , to make it easier to do these casts.

      Using upCast:

      You can do this: 

      
 ns = ints.<Number>upCast();

      In fact, in this particular case, it is even simpler: 

      
 ns = ints.upCast();
